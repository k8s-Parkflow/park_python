# 패키지 구조 리팩터링 진행계획

작성일: 2026-03-12
기준 문서: `docs/refactor/0312/서비스_패키지_구조_개선안.md`

## 현재 상태

- 구조 개선안 작성 완료
- 리팩터링 실행 계획 수립 완료
- `parking-query-service` 파일럿 리팩터링 진행 중

## 리팩터링 목표

- 서비스별 패키지 구조를 `DDD + 간결한 Clean Architecture` 기준으로 통일한다.
- 루트 평면 구조를 줄이고, bounded context 중심 구조로 정리한다.
- 의존 방향을 `interfaces -> application -> domain` 중심으로 고정한다.
- ORM, gRPC, 외부 시스템 의존 구현은 `infrastructure`로 분리한다.

## 전체 진행 순서

1. 공통 구조 기준 확정
2. `parking-query-service` 파일럿 리팩터링
3. `parking-command-service` 리팩터링
4. `orchestration-service` 리팩터링
5. `zone-service` 리팩터링
6. `vehicle-service` 리팩터링
7. 테스트 구조 정리
8. 문서 및 legacy 경로 정리

## 서비스 적용 우선순위

| 순서 | 서비스 | 이유 | 상태 |
| --- | --- | --- | --- |
| 1 | `parking-query-service` | 규모가 적당하고 HTTP/gRPC/ORM/repository가 모두 섞여 있어 파일럿으로 적합 | 진행 중 |
| 2 | `parking-command-service` | 일부 DDD 구조가 이미 있어 기준 구조로 수렴시키기 좋음 | 대기 |
| 3 | `orchestration-service` | 사가 도메인으로 묶으면 구조 단순화 효과가 큼 | 대기 |
| 4 | `zone-service` | 비교적 단순하여 앞선 기준을 그대로 적용하기 좋음 | 대기 |
| 5 | `vehicle-service` | 가장 단순하여 마지막 정리 대상으로 적합 | 대기 |

상태:

- 완료

- 실제 이동 규칙, import 정리 방식, 조립 지점을 확정한다.

작업 순서:

- [x] `parking_view` bounded context 패키지 생성
- [x] `interfaces/http`, `interfaces/grpc` 골격 생성
- [x] `application/use_cases` 골격 생성
- [x] `domain` 포트/엔터티 기준 정리
- [x] `infrastructure/persistence`, `infrastructure/clients` 골격 생성
- [x] 현재 위치 조회 유스케이스 분리
- [x] zone availability 조회 유스케이스 분리
- [x] projection 반영/보상 유스케이스 분리
- [x] HTTP view를 interfaces로 이동
- [x] gRPC application/service를 interfaces + application으로 분리
- [x] repository 구현을 infrastructure로 이동
- [x] gRPC client 구현을 infrastructure로 이동
- [x] bootstrap 또는 factory 조립 지점 도입

### Phase 2. parking-command-service 리팩터링

목표:

작업 순서:

- [ ] `domains/parking_record` 경로를 목표 구조로 승격
- [ ] 루트 `views.py`, `services.py`, `urls.py` 비즈니스 로직 이동

## 서비스별 작업 착수 기준

각 서비스는 아래 조건을 만족할 때 다음 서비스로 넘어간다.

1. 목표 bounded context 패키지가 생성되어 있을 것
2. HTTP/gRPC 진입점이 `interfaces` 하위로 이동했을 것
3. 핵심 유스케이스가 `application` 하위로 이동했을 것
4. repository/client 구현이 `infrastructure` 하위로 이동했을 것
5. 루트 평면 비즈니스 파일이 제거되거나 thin wrapper 수준으로 축소되었을 것
6. 테스트 import 경로가 새 구조를 기준으로 정리되었을 것

## 주의 사항

- 구조 이동 커밋과 로직 변경 커밋은 분리한다.
- 한 서비스 리팩터링 중에는 다른 서비스 구조까지 동시에 건드리지 않는다.
- `shared`는 정말 공통인 경우에만 사용한다.
- 폴더를 통일하는 것이 목적이 아니라, 책임과 의존 방향을 통일하는 것이 목적이다.
