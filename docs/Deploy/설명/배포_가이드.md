# 인프라팀용 쿠버네티스 배포 가이드

## 문서 목적

- 현재 백엔드 코드 구조를 기준으로, 인프라 담당자의 쿠버네티스 배포 구조 설계 시 필요한 정보의 단일 문서 정리.
- "무엇을 어떤 단위로 띄워야 하는가"와 "무엇을 외부에 노출해야 하는가" 설명에 집중한 문서 성격.
- Dockerfile 구현 방식은 인프라 담당 범위로 간주하고, 여기서는 컨테이너 실행 명령과 필요한 환경변수만 정리하는 범위 설정.

## 한 줄 요약

- 외부에는 `orchestration-service` HTTP 만 노출한다.
- 내부에는 `vehicle`, `zone`, `parking-command`, `parking-query` 를 gRPC 서비스로 띄운다.
- 각 서비스는 자기 DB 만 사용한다.
- 배포 순서는 `DB 준비 -> 서비스별 migrate Job -> 내부 gRPC 배포 -> orchestration HTTP 배포 -> Ingress 연결` 이다.

## 현재 백엔드 구조 기준 권장 워크로드

### 외부 공개

- `orchestration-http` Deployment
- `orchestration-http` ClusterIP Service
- `Ingress` 또는 외부 LB 는 `orchestration-http` 만 연결

### 내부 전용

- `vehicle-grpc` Deployment + ClusterIP Service
- `zone-grpc` Deployment + ClusterIP Service
- `parking-command-grpc` Deployment + ClusterIP Service
- `parking-query-grpc` Deployment + ClusterIP Service

### 운영 Job

- `orchestration-migrate` Job
- `vehicle-migrate` Job
- `zone-migrate` Job
- `parking-command-migrate` Job
- `parking-query-migrate` Job

## 왜 이렇게 배포해야 하는가

- 클라이언트 요청의 공식 진입점으로서의 `orchestration-service` HTTP.
- 오케스트레이터의 내부 `vehicle`, `zone`, `parking-command`, `parking-query` 대상 gRPC 통신 구조.
- 각 서비스별 런타임, settings, migrate 명령 분리 완료 상태.
- 내부 서비스 HTTP 런타임의 존재와 별개로, 현재 운영 기준 외부 공개 대상이 아니라는 점.

즉, 인프라 기준 기본 토폴로지로서의 "오케스트레이터 HTTP 1개 + 내부 gRPC 4개".

## 내부 통신 방식

### 오케스트레이터가 참조하는 gRPC target 환경변수

- `VEHICLE_SERVICE_GRPC_TARGET`
- `ZONE_SERVICE_GRPC_TARGET`
- `PARKING_COMMAND_SERVICE_GRPC_TARGET`
- `PARKING_QUERY_SERVICE_GRPC_TARGET`

오케스트레이터 코드의 해당 환경변수 기반 내부 gRPC 서버 연결 구조.

권장 값 예시:

- `VEHICLE_SERVICE_GRPC_TARGET=vehicle-grpc:50051`
- `ZONE_SERVICE_GRPC_TARGET=zone-grpc:50052`
- `PARKING_COMMAND_SERVICE_GRPC_TARGET=parking-command-grpc:50053`
- `PARKING_QUERY_SERVICE_GRPC_TARGET=parking-query-grpc:50054`

## 서비스별 기본 포트 예시

- vehicle gRPC: `50051`
- zone gRPC: `50052`
- parking-command gRPC: `50053`
- parking-query gRPC: `50054`
- HTTP: 기본 `8000`

실제 스크립트는 내부적으로 아래 환경변수를 사용

- HTTP
  - `HTTP_HOST`
  - `HTTP_PORT`
  - `HTTP_WORKERS`
- gRPC
  - `VEHICLE_GRPC_HOST`
  - `VEHICLE_GRPC_PORT`
  - `ZONE_GRPC_HOST`
  - `ZONE_GRPC_PORT`
  - `PARKING_COMMAND_GRPC_HOST`
  - `PARKING_COMMAND_GRPC_PORT`
  - `PARKING_QUERY_GRPC_HOST`
  - `PARKING_QUERY_GRPC_PORT`

## DB 연결 전략

## 전제

- 현재 서비스 settings 기본 구현의 SQLite 경로 기반 구조.
- 즉, 현재 코드 기준 `*_DB_NAME` 환경변수의 "DB 접속 문자열"이 아닌 파일 경로 수신 구조.

서비스별 기본 DB 환경변수:

- `ORCHESTRATION_DB_NAME`
- `VEHICLE_DB_NAME`
- `ZONE_DB_NAME`
- `PARKING_COMMAND_DB_NAME`
- `PARKING_QUERY_DB_NAME`

### 인프라 관점 권장안

- 운영 환경에서의 SQLite PVC 운영보다 외부 RDBMS 또는 managed DB 사용 권장.
- 다만 현재 백엔드 설정은 SQLite 전용이므로, 운영 DB 전환 시 백엔드 설정 확장 추가 필요.

## 최종 정리

- 인프라 기준 1차 배포 단위로서의 `HTTP 1 + gRPC 4 + migrate Job 5`.
- 외부 공개 대상의 `orchestration-http` 단일화.
- 내부 서비스의 ClusterIP 와 gRPC target 환경변수 기반 연결 구조.
- DB 의 서비스별 분리 원칙.
- 현재 백엔드 기준 SQLite 기본값으로 인한, 운영 DB 전략에 대한 인프라와 백엔드 간 별도 합의 필요성.
